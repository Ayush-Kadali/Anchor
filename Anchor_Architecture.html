Serial/SSH Connection Flow

How a browser keystroke reaches a physical device and back

1. User types "hello" in browser
dashboard.jsp JavaScript: ws.send("hello")
↓
2. TerminalWebSocket.onMessage("hello")
@OnMessage fires in Java. Parses command.
↓
3. ConnectionManager.send(sessionId, bytes)
Singleton looks up which Connection belongs to this session.
Checks isOwner() - only owner can send.
↓
4a. SerialConnection.send()
serialPort.writeBytes(data, len)
jSerialComm writes to USB port
4b. SshConnection.send()
outputStream.write(data)
outputStream.flush()
JSch sends over TCP
↓
5. Physical Device / Remote Server
Arduino processes command, or Linux shell executes it.
Sends response back.
↑
6. Reader Thread (background, every 50ms)
connection.receive() polls for data.
If data available: session.getBasicRemote().sendText(data)
Also: ConnectionManager.broadcastToViewers(data)
↓
7. Browser receives data
ws.onmessage fires in JavaScript.
appendTerminal(event.data) displays it.
Key Java Concepts in this flow:
- Polymorphism: ConnectionManager calls send()/receive() without knowing if it's Serial or SSH
- Daemon Thread: Reader thread runs in background, auto-stops when server stops
- volatile: reading flag visible across threads
- byte[]: Raw binary data for hardware communication
- WebSocket: Bidirectional persistent connection (unlike HTTP request-response)

Multi-User Access (Owner/Viewer)

What happens when multiple people connect to the same device

Step 1: Student A connects to COM3
connect serial COM3 9600

ConnectionManager.connectSerial():
  1. Check portToConnection map - is COM3 in use? NO
  2. Create new SerialConnection("COM3", 9600)
  3. serialConnection.connect() - opens port
  4. Store in connections map
  5. Set Student A as OWNER
  6. Start reader thread for this connection
Step 2: Student B connects to COM3 (same device!)
connect serial COM3 9600

ConnectionManager.connectSerial():
  1. Check portToConnection map - is COM3 in use? YES
  2. Port owned by Student A
  3. Add Student B to connectionViewers list
  4. Set Student B as VIEWER
  5. Student B sees output but cannot type commands
  6. Message: "Device in use by Student A. Joined as viewer."
Step 3: Device sends output
Reader thread calls connection.receive()

Data goes to:
  → Student A (owner) via session.getBasicRemote().sendText()
  → Student B (viewer) via broadcastToViewers()
  → Student C (viewer) via broadcastToViewers()

Everyone sees the same output simultaneously.
Step 4: Who can type?
Student A types "ls" → ConnectionManager.send() → checks isOwner() → YES → sent to device

Student B types "ls" → ConnectionManager.send() → checks isOwner() → NO → blocked
Message: "You are a VIEWER. Only the owner can send commands."
Step 5: Student A disconnects
ConnectionManager.disconnect(Student A session):

  1. Student A was OWNER
  2. Check connectionViewers list - Student B is waiting
  3. Promote Student B to OWNER
  4. Send message to B: "You are now the OWNER. You can type commands."
  5. Student B can now control the device

If no viewers: close the SerialConnection, free the port.
Java Concepts:
- synchronized: connectSerial() is synchronized to prevent race condition when two people connect at the same millisecond
- ConcurrentHashMap: Thread-safe map for multiple WebSocket threads accessing simultaneously
- Iterator.remove(): Safely removes dead viewer sessions while iterating
- instanceof: Check if connection is SerialConnection for port cleanup
- Observer pattern: broadcastToViewers() pushes data to all subscribers

Network Architecture

Multiple laptops as servers, accessed from any browser on the network

Laptop A (Server)
Anchor Server running
IP: 192.168.1.10:8080

Devices connected:
- Arduino Mega (COM3)
- Sensor Board (COM5)
Laptop B (Server)
Anchor Server running
IP: 192.168.1.20:8080

Devices connected:
- Cisco Router (COM1)
- Raspberry Pi (USB)
Laptop C (Client)
No Anchor server needed
Just a browser

Can access:
- 192.168.1.10:8080 (A's devices)
- 192.168.1.20:8080 (B's devices)
WiFi Network / LAN (same subnet)

How It Works

1. Each laptop with hardware runs its own Anchor server
cd anchor && mvn jetty:run
Jetty binds to 0.0.0.0 (all network interfaces) - not just localhost
2. Server detects its own hardware
DeviceScannerServlet calls SerialPort.getCommPorts() to list local USB ports.
Only ports physically connected to THIS laptop are shown.
3. Client connects from any browser
Student opens http://192.168.1.10:8080/anchor from their laptop.
Logs in, scans devices, connects to Arduino on Laptop A.
All communication goes through WebSocket over the network.
4. Multiple clients, multiple servers
Student 1 connects to Laptop A's Arduino.
Student 2 connects to Laptop B's Router.
Student 3 connects to Laptop A's Arduino as VIEWER.
Professor monitors all from their own browser.
Configuration (pom.xml):
<httpConnector>
  <host>0.0.0.0</host> <!-- listen on all interfaces -->
  <port>8080</port>
</httpConnector>

0.0.0.0 means "accept connections from any network interface".
Default localhost (127.0.0.1) only accepts connections from the same machine.
This one change makes the entire application network-accessible.
